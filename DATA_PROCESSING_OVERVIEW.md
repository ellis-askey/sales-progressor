## 2. User Roles and Access

There are two distinct authenticated user tiers:

### 2a. Agency Progressor (staff login)
- Created by the agency admin (`role: "admin"`) or by a superadmin
- Has full read/write access to their agency's transactions, contacts, comms, reminders, and files
- Can view the agent portal but does not use it day-to-day

### 2b. Agent Portal User
- Solicitors, vendors, purchasers linked to a specific transaction
- Authenticated separately; can only see their own transaction(s)
- Cannot see other transactions, contacts outside their deal, or agency-wide data
- Multi-tenancy enforced at query level via `agencyId` filter on every DB call

### 2c. Superadmin
- Single internal account (`role: "superadmin"`) — TO CONFIRM: is there more than one?
- Can see all agencies; not a customer-facing role

## 3. Authentication

### Mechanism
- **Provider:** NextAuth.js Credentials provider (username + password)
- **Session strategy:** JWT (not database sessions)
- **Password hashing:** bcryptjs, cost factor 12

### JWT Contents
The JWT issued on login contains:
```
id, name, email, role, agencyId, firmName
```
This data is present in the signed cookie on every request. No personal data beyond this is stored in the JWT.

### Database Session Storage
- `Account` and `Session` tables exist in the Prisma schema (standard NextAuth adapter tables)
- **These tables are unused** — the Credentials provider with JWT strategy does not write to them
- Both tables will always be empty in production

### Password Reset
- A time-limited token is issued and stored in `PasswordResetToken` table (email + hashed token + expiry)
- Tokens expire after **1 hour**
- The reset flow does **not** confirm whether an email address is registered (prevents user enumeration)
- Reset tokens are deleted on use

### Session Expiry
- TO CONFIRM: `maxAge` value set in NextAuth config (default is 30 days if not overridden)

## 7. AI Processing — Anthropic (Claude)

This is the most significant third-party data sharing in the application. Personal data is sent to Anthropic's API in two distinct flows.

### Model Used
**Claude Haiku 4.5** (`claude-haiku-4-5-20251001`) via Anthropic API.

---

### Flow 1: Chase Message Generation

**Trigger:** Progressor clicks "Generate message" in the Chase Drawer.

**Data sent to Anthropic:**
- Property address
- Sale price
- Purchase type (residential / BTL / etc.)
- Tenure (freehold / leasehold)
- Expected exchange date
- Milestone name (e.g. "Mortgage Offer Received")
- Chase count (number of previous chases)
- Last outbound communication text (truncated to 300 characters)
- **Primary contact: full name + role type** (e.g. "David Mitchell — solicitor")
- **Secondary contacts: names + role types** (all contacts on the transaction)
- Progressor's own name (from session)

**Token limit:** 600 output tokens.

**What Anthropic returns:** A draft chase message (plain text).

**Persistence:** The generated text is returned to the browser and pre-filled in the textarea. If the progressor clicks Send, the generated text is saved to `CommunicationRecord.generatedText`. It is NOT automatically saved by the generation API route — only on explicit send.

---

### Flow 2: Email-to-Milestone Parsing

**Trigger:** Progressor pastes a received email into the "Parse email" interface.

**Data sent to Anthropic:**
- Raw email text (up to 3,000 characters — truncated)
- This text is a real email received by the progressor and may contain: solicitor name, firm name, client names, legal references, financial details, and any other content present in the email

**What Anthropic returns:** JSON array of suggested milestone updates (milestone name + new status).

**Persistence:** The raw email text is NOT stored. Only the milestone updates actioned by the progressor are saved to the database.

## 9. Property Intelligence APIs

### 9a. Land Registry SPARQL (open public endpoint)

- **URL:** `https://landregistry.data.gov.uk/landregistry/query`
- **Purpose:** Retrieve price paid history for a property
- **Data sent:** Postcode + house number or street name
- **Authentication:** None required (public endpoint)
- **Data returned:** Historical sale prices, dates, buyer/seller type (individual/company) — this is publicly registered data
- **No personal data** is sent beyond property location identifiers

### 9b. EPC Register API (Government, credentialed)

- **URL:** `https://epc.opendatacommunities.org/api/v1/domestic/search`
- **Purpose:** Retrieve Energy Performance Certificate data for a property
- **Data sent:** Postcode + building number
- **Authentication:** `EPC_API_KEY` environment variable (government-issued API key)
- **Data returned:** EPC rating, energy efficiency scores, property characteristics — publicly registered data
- **No personal data** is sent beyond property location identifiers

## 14. Security Measures

### In place
- **Password hashing:** bcryptjs cost 12 (strong; equivalent to ~300ms hash time)
- **JWT signing:** NextAuth signs JWTs with `NEXTAUTH_SECRET`; tokens cannot be forged
- **Multi-tenancy isolation:** `agencyId` filter applied to all data queries in application code
- **HTTPS:** Enforced by Vercel (all traffic in transit encrypted)
- **User enumeration prevention:** Password reset does not confirm whether email is registered
- **Supabase connection:** Via Prisma with server-side env vars; credentials never exposed to browser

### Not in place
- **Rate limiting:** No rate limiting on any API route — including authentication, AI generation, and email sending
- **Row-level security (RLS):** Not configured in Supabase; multi-tenancy relies solely on application code
- **Input validation:** TO CONFIRM — Zod or equivalent schema validation not observed in API routes reviewed; some routes trust client-supplied `transactionId`, `chaseTaskId` etc. without verifying ownership
- **Content Security Policy:** TO CONFIRM — not observed in `next.config.*`
- **GDPR deletion:** No routes or UI for data subject deletion requests
- **GDPR export:** No routes or UI for data subject access requests (SARs)
- **Audit log:** No immutable audit trail of who accessed or modified data (beyond `updatedAt` timestamps)
