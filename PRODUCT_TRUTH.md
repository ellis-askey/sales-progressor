# PRODUCT_TRUTH.md

*Codebase audit — read-only. Written for the founder to use as a reference when drafting marketing copy and onboarding emails. All claims below are grounded in the actual code.*

---

## 1. Elevator pitch

Sales Progressor is a web platform for UK residential estate agencies that tracks property transactions through a structured milestone checklist, alerts the agency team when chasing is due, lets buyers and sellers check their progress via a branded client portal, and automatically sends a weekly check-in email to clients who have heard nothing for seven days. An AI model (Claude Haiku) can draft chase messages on request and labels problem files overnight. The system does not negotiate with solicitors, send emails autonomously on behalf of the agency, or replace the human progressor — it surfaces what needs doing and records what was done.

## 2. Core mechanics (plain English)

### Milestones

Milestones are defined centrally in the database as templates. There are 47 in total: 20 on the vendor (seller) side, coded VM1–VM20, and 27 on the purchaser (buyer) side, coded PM1–PM27. Each definition carries an `orderIndex`, a `blocksExchange` flag, and an optional `eventDateRequired` flag. When a transaction is created, 47 per-transaction completion records are created, each starting in one of three states: `locked`, `available`, or `not_required` (the last is set automatically at creation for milestones that don't apply — e.g. mortgage milestones for cash buyers, management pack milestones for freehold properties).

A milestone moves from `locked` to `available` when all of its direct prerequisites are marked `complete` or `not_required`. This is evaluated in code whenever any milestone is confirmed, and the system automatically unlocks the next milestone(s) in the chain. A milestone moves from `available` to `complete` when a progressor, agent, or (for certain milestones) the client confirms it through the portal.

`blocksExchange: true` on a milestone definition means that milestone must be either `complete` or `not_required` before the "solicitor ready to exchange" gate milestone (VM18 for vendor, PM25 for purchaser) is unlocked. The system literally prevents those gate milestones from becoming available — they stay `locked` — until every same-side blocking milestone is done. Exchange itself (VM19/PM26) can only be confirmed after both gate milestones are confirmed.

### Reminders

There are 47 reminder rules seeded in the database, one per milestone. Each rule records:

- An anchor milestone (the event that starts the clock, or the transaction creation date if no anchor is set)
- `graceDays` — how many days after the anchor before chasing begins
- `repeatEveryDays` — how often the chase task escalates if unanswered
- `escalateAfterChases` — how many chases before the task is marked escalated (priority: escalated)

The reminder engine runs every day at 07:00 UTC via a Vercel Cron job. For each active transaction it checks each rule: if the anchor milestone is complete, it creates or updates a `ReminderLog` record with a calculated due date. If the target milestone is already done, it deactivates the log. Some reminder rules are exchange-gated: they only activate once the transaction is fully exchange-ready.

### Chase tasks

When a `ReminderLog` reaches its due date and has no pending `ChaseTask`, the engine creates one automatically. The chase task is what the progressor or agent sees in their task list. It records how many times it has been chased (`chaseCount`), its priority (`normal` or `escalated`), and whether a communication has been logged against it.

Chase tasks differ from manual tasks: chase tasks are generated by the reminder engine against specific milestone-chasing rules; manual tasks are created freely by any user, can be linked to a transaction or not, and can be created as agent-to-progressor requests from the agent portal.

### Exchange gate

"Exchange-ready" is computed by checking that every milestone definition with `blocksExchange: true` on both the vendor and purchaser sides has a completion record in state `complete` or `not_required`. Only when that is true are VM18 (vendor ready to exchange) and PM25 (purchaser ready to exchange) unlocked. Both gate milestones must then be confirmed before the exchange milestone (VM19/PM26) becomes available.

### Time-sensitive milestones

Certain milestone definitions carry `eventDateRequired: true`. When a progressor confirms one of these milestones, the UI requires them to enter a specific date (e.g. lender valuation date, survey date, mortgage offer date, exchange date, completion date). The date is stored as `eventDate` on the completion record. Some reminder rules use the `eventDate` rather than the `completedAt` date as their anchor — this keeps chasing timed to the actual event rather than the date it was recorded.

Exchange (VM19/PM26) and completion (VM20/PM27) milestones are bilateral: confirming one automatically confirms its counterpart on the other side.

### What happens at exchange — the reconciliation sweep

When a progressor confirms exchange (VM19 or PM26) on a transaction where some milestones are still outstanding, the system shows a modal listing every incomplete milestone. The progressor can assign dates to each one. The system then:

1. Sweeps all outstanding milestones to `complete` (marked with `reconciledAtExchange: true`) in a single database transaction
2. Cancels all pending chase tasks for those swept milestones
3. Marks the associated reminder logs as `completed`
4. Confirms the primary exchange milestone and its bilateral counterpart

This means the milestone list is always clean and accurate after exchange, even if some steps were skipped or confirmed late.

### Agent portal vs progressor app

**Client portal** (no login required — access via a unique token link):
- See their own side's milestone checklist (vendor sees vendor milestones; purchaser sees purchaser milestones)
- See percentage progress and key dates
- Confirm certain milestones themselves (only those in `available` state)
- Send a freetext message to the progressor
- Receive replies from the progressor in the same thread
- View a timeline of outbound communications marked `visibleToClient`
- Subscribe to browser push notifications for progress updates

**Agent portal** (requires login, role: negotiator or director):
- See only their own files (or all agency files for directors, depending on configuration)
- View the milestone list and progress for each file
- See escalated chase tasks and overdue reminders across their files
- Create manual tasks / requests to the progressor
- View weekly briefing email (delivered Monday 07:00)
- Access Land Registry price paid history and EPC data for each property (on demand, via a button that queries the Land Registry SPARQL API and EPC Open Data Communities API)

**What agents cannot do in the portal (only progressors can):**
- Confirm milestones
- Send chase emails
- Mark milestones as not-required
- Access other agencies' files
- See the full communication log

### AI features

Two AI calls use Claude Haiku (claude-haiku-4-5-20251001 via the Anthropic API):

**1. Chase message drafting** (`/api/ai/generate-chase`): A progressor opens a pending chase task and clicks "Draft with AI". The system sends the following to Claude: property address, sale price, purchase type, tenure, exchange target date, the milestone being chased (name, side, whether it blocks exchange), the chase count, days since last chase, a snippet of the last outbound message, and the names of the relevant contacts. The progressor chooses a tone (Friendly, Professional, Polite Yet Firm, Chase Up, Urgent, or Final Reminder) and channel (email or WhatsApp). Claude returns a draft message. The progressor reads it, edits if needed, and clicks Send. The AI does not send the message — the progressor does. The draft is recorded in the communication log with `wasAiGenerated: true`.

**2. Problem flag descriptions** (`lib/services/problem-detection.ts`): A nightly cron job (`/api/cron/detect-problems`, 03:00 UTC) applies seven rule-based checks to every active and on-hold transaction: no communication in 10+ days, milestone pace 25%+ behind benchmark, pending chase task 7+ days overdue, exchange approaching with fewer than 25 milestones done, on hold for 14+ days, no portal activity after 14 days, no new milestone in 21+ days. For each newly detected flag, a short plain-English description (max 12 words) is generated by calling Claude with the flag type and its numeric context. This description is stored in `TransactionFlag.reason` and shown in the agent weekly brief and the progressor dashboard.

⚠ The email parsing AI feature (`/api/ai/parse-email`) is disabled. The route returns a 503 with the comment: "privacy review in progress — the email parser sends third-party correspondence (solicitor emails) to Anthropic without the third party's knowledge."

## 3. Automatic vs manual

| Action | Classification | Notes |
|--------|---------------|-------|
| Milestone emails to clients when a milestone is confirmed | **Automatic** | Fired server-side immediately after any milestone is marked complete, without any human action. Uses per-milestone email copy from `portal-copy.ts`. |
| Weekly update emails to clients | **Automatic** | Cron job runs Sunday 18:00 UTC. Skips any transaction that already had outbound communication in the past 7 days. |
| Weekly brief emails to agents | **Automatic** | Cron job runs Monday 07:00 UTC. Sent to all users with role `negotiator` or `director`. |
| Surfacing overdue tasks to the progressor | **Automatic** | The reminder engine (daily 07:00 UTC) creates chase tasks when logs become due. They appear in the progressor's task list without any action required. |
| Creating chase tasks | **Automatic** | Created by the reminder engine when a reminder log is due and has no pending task. Also created lazily when a progressor or agent loads the reminders view for a transaction. |
| Sending chase emails | **Manual** | A human clicks Send every time. The AI can draft the text, but the progressor must review and manually submit the message via `/api/chase/send-email`. Nothing sends autonomously. |
| Identifying which transactions need attention | **Automatic** | The problem detection cron (03:00 UTC) detects and stores flags. They appear on dashboards without human input. |
| Blocking exchange if a blocking milestone is incomplete | **Automatic** | The exchange gate milestones (VM18/PM25) stay locked in the database until all blocking milestones are done. The UI enforces this — a progressor cannot confirm exchange while any blocker is outstanding. |
| Generating the survey email at completion | **Triggered-manual** | The survey email is sent automatically when a progressor marks a transaction status as `completed`. The trigger is human (status change), but the send is immediate and automatic. |
| Portal invite email | **Manual** | A progressor or admin clicks "Send invite" on the contacts panel. The `generatePortalTokenAction` creates the token; the invite email is then sent on explicit button press (`/api/portal/invite`). |
| AI-generated chase suggestion | **Manual** | Requires a progressor to open a task, choose a tone, choose a channel, and click "Generate". The result is a draft only. |
| Logging email interactions | **Mixed** | Automated milestone emails and weekly updates are logged automatically (`isAutomated: true`). Chase emails sent manually are logged automatically on send. Portal messages are logged automatically. Manual notes (phone calls, inbound correspondence) must be entered by the user. |

## 4. What the system does NOT do

- **CRM integration**: None. There is no connection to any third-party CRM (Salesforce, HubSpot, Reapit, etc.). Transactions are created manually inside the app.

- **Land Registry data (automatic)**: Land Registry price paid history is fetched on demand when a progressor loads the property intel tab. It is not ingested automatically and is not stored in the database — it is fetched live each time and cached by Next.js for 24 hours. The data comes from the Land Registry SPARQL endpoint (public API, no auth required).

- **EPC data (automatic)**: EPC data is also fetched on demand from the EPC Open Data Communities API, not stored, and cached for 24 hours. Requires `EPC_API_EMAIL` and `EPC_API_KEY` environment variables to be set. If they are not, EPC data silently returns null.

- **SMS / WhatsApp (sending)**: The system can draft a WhatsApp-formatted message and the email template includes a WhatsApp CTA button, but there is no WhatsApp Business API integration. WhatsApp messages cannot be sent from within the platform. The channel field on a communication record can be set to `whatsapp`, but this is a manual label — the platform does not route anything through WhatsApp.

- **SMS**: Same position — `sms` is a valid `CommMethod` enum value for labelling a record, but the platform cannot send SMS.

- **Phone call logging**: A phone call can be logged as a communication record with `method: phone`, but there is no dialler, no call recording, and no automatic logging.

- **Revenue / cash flow forecasting**: No such feature exists. The schema stores `agentFeeAmount` and `agentFeePercent` per transaction, but there are no reports, dashboards, or projections built around fee data.

- **Mortgage lender integration**: None. The mortgage application and offer milestones are tracked as steps in the checklist; no API connection to any lender.

- **Agent billing / invoicing**: Not present. The platform does not generate invoices for agents or track payments to the agency.

- **Two-way email sync**: The platform cannot read email replies. Inbound correspondence must be manually logged by the progressor. The email parse feature that would have used AI to log solicitor emails has been disabled pending a privacy review.

- **Automatic transaction creation**: Transactions are created manually by a progressor or admin inside the app. There is no integration with a portal submission form, Rightmove, or any other external source that would create a transaction automatically.

## 5. Common misconceptions — things NOT to write in copy

> "Sales Progressor removes manual chasing" — *Wrong because: the system tells the progressor who to chase and when, but every chase email still requires a human to review the draft and click Send. Nothing is sent autonomously.*

> "AI handles your client communications" — *Wrong because: the AI drafts a single message on request. The progressor reads it, edits it, and sends it manually. The AI has no ability to initiate or send communications independently.*

> "Clients get real-time updates" — *Wrong because: clients are notified by email and push notification when a milestone is confirmed, but those notifications depend on the progressor or client confirming milestones in the first place. There is no live feed pulling data from solicitors' systems.*

> "Land Registry and EPC data is automatically pulled for every file" — *Wrong because: it is fetched on demand when a user loads the property intel tab. It is not stored in the database, and EPC lookups require API credentials to be configured.*

> "The platform integrates with your CRM" — *Wrong because: there is no CRM integration. Transactions must be created manually.*

> "WhatsApp chasing is built in" — *Wrong because: the AI can format a message for WhatsApp and the chase UI labels the channel, but there is no WhatsApp API. The user copies the text and sends it themselves.*

> "The system blocks exchange automatically until all checks are complete" — *Mostly true but needs precision: the gate milestones (VM18/PM25) are locked in the system until all blocking steps are done — but a progressor could still manually mark VM18 available via database intervention. The enforcement is at the application layer, not at a legal or regulatory level.*

> "Clients can track both sides of their transaction" — *Wrong because: clients only see their own side. A vendor sees vendor milestones; a purchaser sees purchaser milestones. Neither sees the other side.*

> "The weekly update email is personalised with their latest progress" — *Partially wrong: the weekly email is a generic check-in ("no news is good news") message, not a tailored update. It does include the exchange target date if one is set, but it does not list recently completed milestones or outstanding steps.*

> "Sales Progressor reads and logs your email inbox automatically" — *Wrong because: the email parsing feature is disabled. All inbound correspondence must be manually logged.*

> "Portal invites are sent automatically when a transaction is created" — *Wrong because: the portal token is created at transaction creation, but the invite email is only sent when a user manually clicks "Send invite" on the contacts panel.*

> "The AI detects problems proactively and alerts you immediately" — *Partially wrong: problem detection runs once per day at 03:00 UTC, not in real time. Flags appear the morning after the threshold is crossed, not at the moment it happens.*

## 6. What's true that the founder might not realise

**Clients can confirm milestones themselves.** Buyers and sellers can log into their portal and tick off certain steps — for example, confirming they have instructed their solicitor or returned signed contracts. When they do, the progressor gets an immediate email notification, and the other party also gets a progress update email automatically. This is a genuinely differentiated feature: clients are participants, not just spectators.

**Every outbound email is automatically logged to the transaction timeline.** Whether it is a milestone notification, a weekly update, a portal invite, or a manually sent chase email, a `CommunicationRecord` is written to the database with `isAutomated` flagged where appropriate. This means the progressor always has a complete audit trail without any manual effort.

**The reminder engine knows about the relationship between milestones.** Rules are anchored to specific predecessor milestones, not just to a fixed calendar date. For example, the "chase: initial replies received by buyer's solicitor" rule does not start counting until the seller's solicitor has issued its initial replies (VM12). This means chasing is contextually correct — it does not fire for steps that are not yet relevant.

**Escalation is automatic.** A chase task that is not resolved within the configured repeat window has its `chaseCount` incremented and its priority changed to `escalated` by the engine. Escalated files are highlighted separately in the agent weekly brief and in the progressor's task view. The agent does not have to manually flag a stalled file.

**The agent portal includes a "proactive alerts" section in the weekly brief.** The Monday brief email sent to agents includes any AI-detected flags (long silence, stalled progress, unanswered chase, exchange approaching with gaps) for their files. Agents see these even if no one has told them about them manually.

**The system supports property chains.** The schema has a `PropertyChain` model with `ChainLink` records. Progressors can link multiple transactions into a chain and track each position. This is built but is not heavily marketed.
