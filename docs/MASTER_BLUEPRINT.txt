What milestones are in this system
Milestones are not just labels on a progress bar.
In Sales Progressor, a milestone does three jobs:
1. it shows where the transaction is
2. it helps control what should happen next
3. it acts as a trigger for reminders, tasks, and exchange readiness logic
So the milestone system needs to be treated as core business logic, not just UI.

Proposed milestone model
1. Milestone Definitions are templates
A Milestone Definition is the master version of a milestone.
Examples:
* Solicitor instructed
* Draft contract received
* Searches ordered
* Enquiries raised
* Mortgage offer issued
* Signed contracts returned
* Deposit transferred
* Ready to exchange
These live at the system level and define:
* name
* side
* order
* whether it blocks exchange
* whether it is time sensitive
They are not tied to one file until a completion is created.

2. Milestone Completions are transaction specific
A Milestone Completion is the record that says:
this milestone has been completed on this transaction
So:
* the definition is the template
* the completion is the instance on a real file
This is the right model because a milestone can be completed:
* once on one file
* later on another file
* deactivated/reactivated if needed

3. Vendor and purchaser milestones stay separate
This is non negotiable.
Even if milestone names feel similar, vendor side and purchaser side must remain separate because:
* they progress differently
* they trigger different reminders
* they can block exchange independently
* they can complete at different times
So the system should never treat “signed contracts returned” as one universal milestone unless it is explicitly separated by side.
That means:
* Vendor signed and returned contracts
* Purchaser signed and returned contracts
are separate definitions.

Completion behaviour
4. Milestones are completed by creating a completion record
When a user marks a milestone complete, the system should create a Milestone Completion record.
That record should include:
* transaction
* milestone definition
* active state
* completed at timestamp
* optional completed by user
So the milestone itself is not “ticked” on the definition.
A completion record is created against the file.

5. Completions must use active/inactive state, not hard deletion
This is really important.
If a user re-clicks, reverses, or corrects a milestone, we should not delete history.
Instead:
* old completion can become inactive
* new completion can become active if needed
Why this is right:
* preserves audit trail
* avoids weird hidden data loss
* makes reminder logic safer
* mirrors how your Bubble logic already works
So for any transaction + milestone definition:
* there may be multiple completion records over time
* but only one should normally be active

6. Active completion is the live truth
For system logic, the only completion that counts is the active one.
That means:
* progress state
* prerequisite logic
* reminder triggers
* exchange checks
should use:
active completion exists = yes/no
Inactive completions are history, not current truth.

7. Completion timestamps matter
Every active completion should store when it was completed.
This matters because:
* some reminders are based on completion timing
* reporting/timeline views need chronology
* time sensitive follow ups may depend on it
So completed_at is not optional in practice.

Sequence and order
8. Milestones should have an order index
Each milestone definition should have an order_index within its side.
This does not mean the system blindly forces strict linear progression.
It does mean:
* UI can display the progression logically
* system can reason about “before” and “after”
* reminders can use preceding milestone concepts
* exchange checks can understand what stage the file is broadly at
So order exists for structure, but not every later milestone must require every earlier milestone unless explicitly defined.
That distinction matters.

9. Order is guidance, not the only logic
This is important because property transactions are messy.
A file may not move in a perfectly tidy sequence every time.
So:
* order helps orient the system
* but true logic should rely on rules and completions, not just “everything above must be done”
Otherwise you get false blocks and weird exceptions.

Availability and prerequisites
10. Some milestones may have prerequisite logic
A milestone may only become meaningfully available once another milestone is complete.
Example:
* you would not normally complete “draft contract received” before “solicitor instructed”
So each milestone may optionally have:
* prerequisite milestone(s)
This is not necessarily about hard blocking the UI at first.
It may simply be used to:
* validate workflow
* power logic
* flag odd states
* guide what can trigger reminders
We should decide later whether prerequisites are:
* soft guidance only
* or hard restrictions in the UI
My recommendation for V1:
store the logic, but don’t over-police the UI initially
That keeps the system practical.

What this means in practice

Rule 1

Rule 2

Rule 3

Proposed exchange gating model
1. blocks_exchange lives on milestone definitions
Each milestone definition can carry a boolean flag:
* blocks_exchange = yes
* blocks_exchange = no
This means:
if this milestone is still incomplete on the transaction, it counts as something that prevents the file being considered ready to exchange
This is the cleanest approach because it keeps exchange logic tied to the milestone structure itself, not buried in random conditions elsewhere.

2. Exchange readiness is not a manual status first
The system should not treat “ready to exchange” as just a free-floating status someone manually selects.
It should primarily be a derived state based on milestone completion logic.
Manual override may exist later if you want one, but the core truth should be:
the file is ready to exchange when all required exchange-blocking milestones that apply have active completions
That is much safer and far more consistent.

3. Exchange gating should check both sides of the transaction
A transaction is only genuinely ready to exchange when the required blocking milestones on:
* vendor side
* purchaser side
* relevant agent side, if any
have all been completed.
It cannot just look at one side.

4. Only active completions count
Like everything else in this system:
* active completion = counts
* inactive completion = history only
So when checking whether a blocking milestone is satisfied, the system must check for an active milestone completion.

5. Not every milestone should block exchange
This is really important.
There will be milestones that are useful for tracking and workflow, but should not be part of the exchange gate.
Examples of likely exchange blockers:
* signed contracts returned
* deposit transferred
* solicitor instructed
* draft contract pack received
* key legal / mortgage / enquiry milestones where relevant
Examples of things that may not block exchange:
* informational admin steps
* internal housekeeping
* later completion-only items
* milestones that are helpful but not essential to exchange readiness
So blocks_exchange needs to be selective, not broad by default.

6. Exchange and completion milestones themselves should be excluded from the blocking test
This is a key rule.
When the system checks whether the file is ready to exchange, it should not require:
* exchange itself
* completion itself
* any milestone that only happens after exchange readiness
Otherwise you get nonsense circular logic:
“You are not ready to exchange until exchange is complete.”
So the rule should be:
readiness is determined by all relevant blocking milestones that occur before exchange, excluding exchange and completion milestones themselves
This matches how you described it before.

7. “Ready to exchange” reminder logic should only unlock once the gate is passed
The system should not generate exchange-readiness reminders too early.
So for reminders like:
* confirm solicitor is ready to exchange
* exchange date agreement prompts
* related pre-exchange follow ups
there should be a condition like:
only generate once all non-exchange blocking milestones are satisfied
That way those reminders only appear when the file is actually in the right stage.

8. Readiness should be recalculable, not one-time
This is also important.
A file may become ready to exchange, then later something changes:
* a milestone is reversed
* a correction is made
* something is re-opened
* data is adjusted
So “ready to exchange” should not be a one-time permanent event unless explicitly locked.
It should be a derived/recomputable state the system can evaluate whenever needed.
That gives you safer logic and avoids drift between reality and system state.

9. “Ready to exchange” can still have a milestone or reminder layer
This part matters for UX.
Even though readiness is derived, you may still want:
* a visible “ready to exchange” milestone
* a reminder rule tied to readiness
* UI sections that say “Exchange ready”
That is fine.
But the important design principle is:
the visible label is downstream of the gate, not the source of truth for the gate
In other words:
* derived state first
* visible operational marker second

Practical rule set
Exchange readiness rule
A transaction is considered ready to exchange when:
1. all milestone definitions flagged blocks_exchange = yes
2. that are relevant to the transaction
3. and occur before exchange/completion
4. have an active milestone completion
That is the cleanest core rule.

Reminder unlock rule
Reminder rules related to exchange readiness should only generate when:
* exchange readiness condition = true
This includes things like:
* confirm solicitor is ready to exchange
* start exchange coordination
* pre-exchange date alignment tasks

Reversal rule
If a blocking milestone that was previously complete becomes inactive again, the file should no longer be treated as ready to exchange until the condition is restored.
That keeps the state honest.

because the follow up should not be based on:
the day someone ticked “booked”
It should be based on:
the actual valuation date or survey date
That is the whole point of time sensitivity.

2. Completion date and event date are not the same thing
For time-sensitive milestones, we need to store both:
* completed_at = when the user marked the milestone complete
* event_date = when the real-world booked event is taking place, or took place
These must remain separate.
Because someone might confirm on Monday:
“valuation booked”
but the valuation itself might be on Friday.
If the system uses Monday, the reminder comes too early.
If it uses Friday, it works properly.

3. Time-sensitive logic should live on the milestone definition
The milestone definition should carry:
* time_sensitive = yes/no
This tells the system:
this milestone requires event-based logic, not just completion-based logic
That way the behaviour is driven by the milestone type, not improvised per file.

4. Time-sensitive milestones should require an event date when completed
If a milestone definition is marked time_sensitive = yes, then when a user completes it, the UI should require:
* event date
* optionally event notes if useful later
So for example:
* user marks “Mortgage valuation booked” complete
* system asks for the actual valuation date
That becomes part of the completion record.
This should be a hard requirement for those milestone types.

5. Reminder generation should use event date for time-sensitive milestones
This is the main rule.
For time-sensitive milestones:
* reminder due dates should be based on event_date
* not on completed_at
So if your rule is:
create a chase 7 days after the valuation date
the system should calculate:
event_date + 7 days
not:
completion date + 7 days

6. Event date can be future dated
This is important.
A time-sensitive milestone may be completed today even though the event is in the future.
Example:
* survey booked today
* survey happening next week
That is fine.
The milestone can still be complete, because the booking has happened.
But the logic that follows should wait for the actual event date.
So the system must be comfortable with:
* active completion exists
* event date is in the future
That is not an error.

7. If event date changes, downstream due dates should update
Another important rule.
If someone changes:
* valuation date
* survey date
then any downstream reminder logic that depends on that date should be recalculated.
Otherwise the system becomes stale and starts chasing at the wrong time.
So for time-sensitive milestones:
* date edits must trigger a recalculation of dependent next due dates

8. Time-sensitive milestones can still count as complete
This is subtle but important.
A booked event milestone should still count as:
* complete for progress display
* complete for any logic where “booking confirmed” is the relevant state
It is only the date-driven follow up logic that should use event_date.
So:
* completion truth = booking exists
* scheduling truth = actual event date
That distinction keeps the system sane.

9. Not every date field should become time-sensitive
We should keep this narrow and deliberate.
Only milestones where the real-world event date matters to follow-up logic should use this pattern.
Otherwise the system gets overcomplicated.
For now, the obvious examples are:
* mortgage valuation booked
* level 2 survey booked
* level 3 survey booked
Later you can expand if needed.

Rule 2

Rule 3

Rule 4

Rule 5

Proposed reminder model
1. Reminder Rules are templates, not live reminders
A Reminder Rule is a system definition that explains:
when a certain condition is met, create or update a reminder for this kind of follow up
So a Reminder Rule is not the thing a user acts on directly.
It defines:
* what milestone or event triggers it
* what milestone or situation it relates to
* how many days after the trigger it should become due
* whether it depends on exchange readiness
* whether it uses a time-sensitive event date
This keeps the logic reusable and consistent.

2. Reminder Logs are live transaction-specific reminder instances
A Reminder Log is the actual reminder instance created for a specific transaction from a Reminder Rule.
So:
* Reminder Rule = template
* Reminder Log = live instance on a file
That means a Reminder Log should carry:
* transaction
* reminder rule
* next due date
* status
* maybe last evaluated/generated timestamp later if useful
* audit trail/history eventually
This is the right split because the same rule can apply to hundreds of files, but each file needs its own live state.

3. Reminder Rules should connect anchor and target logic
A reminder rule should usually represent something like:
once anchor milestone/event happens, create follow up logic relating to target milestone or target action
Examples:
* once solicitor instructed is complete, set reminder for ID/AML checks
* once valuation booked is complete, set reminder 7 days after the valuation date
* once contracts signed and deposit transferred are complete, allow ready-to-exchange reminder flow
So each rule should be able to express:
* anchor
* target
* delay
* condition flags
That matches your current model well.

4. Anchor drives due date creation
The anchor is the thing that starts the rule.
That could be:
* a milestone completion
* a time-sensitive event date
* an exchange-ready state
In normal cases:
* anchor milestone is completed
* due date becomes anchor date + delay
In time-sensitive cases:
* anchor milestone is completed
* due date becomes event date + delay
In exchange-ready cases:
* due date becomes active only once readiness condition is true

5. next_due_date belongs on the Reminder Log
The live due date for a reminder should sit on the Reminder Log, not only be calculated ad hoc every time.
Why:
* it is easier to display
* easier to filter
* easier to generate chase tasks from
* easier to audit changes
* easier to understand why something is due
So the system should calculate the due date and store it on the Reminder Log.

6. Reminder generation should be idempotent
This is one of the biggest rules in the whole app.
The daily process must be safe to run repeatedly without creating duplicates or corrupting state.
That means:
* if a relevant Reminder Log already exists and is active, the system updates or leaves it alone
* it does not blindly create another one
* if the due date changes, it updates the existing live log where appropriate
* if the rule is no longer valid, it can deactivate/cancel the log depending on the scenario
In plain English:
running the job twice should not create double reminders

7. One live Reminder Log per transaction + rule + active state
For a given transaction and reminder rule, there should normally be only one live active Reminder Log at a time.
History can exist:
* old inactive logs
* cancelled logs
* completed logs
But the live system should not have:
* multiple active logs representing the same current reminder condition
That is the cleanest anti-duplicate rule.

8. Reminder status should be explicit
A Reminder Log should have a status like:
* active
* completed
* cancelled
* inactive
We can finalise naming later, but the idea is:
* active = live and relevant
* completed = resolved
* cancelled/inactive = no longer relevant or superseded
This matters for clean filtering and auditability.

9. Reminder Rules can be gated by exchange readiness
Some reminder rules should only exist once a file becomes exchange ready.
Example:
* confirm solicitor is ready to exchange
These rules should not generate early just because some milestone happened.
So a Reminder Rule should support a flag like:
* requires_exchange_ready = yes/no
If yes:
* the rule only becomes valid once the derived exchange-readiness condition is true
This maps neatly to your existing approach.

10. Reminder Rules must respect “not required”
If a milestone or condition has been explicitly marked:
* not required
then the reminder engine must treat that as satisfying the relevant condition where appropriate.
Example:
* deposit transferred is marked not required because funds are from sale proceeds
* exchange gating should still pass if all other relevant blockers are satisfied
That means reminder evaluation cannot just check:
completion yes/no
It needs to check:
complete OR explicitly not required

11. Time-sensitive rules must use event date, not completion date
This follows directly from what we already defined.
If the anchor milestone is time-sensitive, then:
* due date = event_date + delay
* not completed_at + delay
So Reminder Rules should either:
* infer this from the milestone definition
or
* explicitly support date source
For clarity, I’d say:
the system infers it from the anchor milestone definition.

12. Reminder Rules should be auditable and understandable
A user or admin should later be able to understand:
* why a reminder exists
* what triggered it
* what date it used
* when that due date changed
So even if we do not expose every detail in V1 UI, the system design should preserve that traceability.
That means a Reminder Log should eventually be able to show:
* source rule
* source anchor
* calculated due date
* current status
* last updated reason

Proposed reminder generation workflow
1. The system should have a scheduled recurring job
There should be a recurring background job that evaluates reminder rules across active transactions.
This should be:
* automated
* central
* not dependent on users opening pages
* not dependent on front-end events firing
This replaces the fragile feeling of UI-led automation.
For now, the exact time does not matter architecturally, but the principle does:
reminder evaluation runs from a scheduled backend job

2. The job should evaluate active transactions only
The reminder generation process should only evaluate transactions where reminder logic is still relevant.
Normally this means:
* active transactions
* maybe on hold transactions depending on your chosen behaviour later
* not completed
* not withdrawn
My recommendation for V1:
* active only
* optionally pause on-hold files unless you later choose otherwise
That keeps the logic clean.

3. The job should evaluate reminder rules against each transaction state
For each relevant transaction, the job should:
* inspect milestone completion state
* inspect not required state
* inspect exchange readiness
* inspect event dates where relevant
* inspect existing reminder logs
* determine whether each rule is:
o valid and active
o not yet valid
o due for update
o no longer valid
This means the reminder engine is really:
transaction state + rule definitions ? live reminder log state

4. Reminder Logs should be created before Chase Tasks
This is important.
The system should not create chase tasks directly from raw milestone logic.
Instead:
1. evaluate rule
2. create or update Reminder Log
3. if Reminder Log is due/eligible, create Chase Task from it
That is cleaner because Reminder Logs become the stable middle layer between:
* system logic
* human tasks
This also makes auditing much easier.

5. Reminder Logs should be created or updated, not blindly recreated
For each transaction + rule:
if no active log exists and rule is now valid
* create a new active Reminder Log
* set next_due_date
if active log exists and rule is still valid
* leave it alone if date/state unchanged
* update it if source date or rule condition changed
if active log exists and rule is no longer valid
* deactivate it
This is the idempotent behaviour we want.

6. next_due_date should come from the correct date source
When calculating next_due_date:
normal milestones
use:
* completed_at + delay_days
time-sensitive milestones
use:
* event_date + delay_days
exchange-ready rules
use:
* the date readiness became true, or immediate due logic depending on how we want those handled
We should decide one subtle point there:
For exchange-gated rules, do you want:
* due immediately when ready
or
* due ready date + delay
My guess is:
most of these should be immediate or same day, unless the rule says otherwise.

7. Chase Tasks should be created from eligible Reminder Logs
A Chase Task should be created only when a Reminder Log is due and actionable.
That means the logic should be something like:
* Reminder Log is active
* next_due_date <= today
* no current open Chase Task already exists for that live reminder state
Then:
* create Chase Task
This keeps tasks from being created too early.

8. Only one open Chase Task per live reminder state
Just like reminder logs, tasks also need duplicate protection.
For a given active Reminder Log:
* there should normally be only one current open Chase Task representing that live reminder state
If the existing open task is still active, do not create another.
If it has been completed and the reminder remains live and later becomes due again, then a new task may be created depending on recurrence model.
For now, the safest V1 rule is:
* one open task per active live reminder state

9. Overdue is derived from due date and task status
A reminder or task becomes overdue when:
* it is still active/open
* and the due date has passed
So overdue is not a separate magical state.
It is derived from:
* current status
* due date vs today
This keeps things consistent and easy to reason about.

10. Rule invalidation deactivates live Reminder Logs and should also close the task path
If a Reminder Log becomes invalid because:
* milestone reversed
* exchange readiness lost
* rule conditions no longer met
* milestone marked not required changes logic
then:
* active Reminder Log should be deactivated
* any linked open Chase Task should either be deactivated/cancelled or marked no longer relevant
My recommendation:
use a clean inactive/cancelled state, not silent disappearance.
That preserves traceability.

This is a decision worth locking now or soon.

* if transaction status = on_hold
* reminder generation pauses
* existing active logs/tasks can remain visible but not generate new operational churn unless manually resumed
That usually matches real-life expectations better than continuing to chase everything on hold.

12. Reminder generation should produce audit/activity entries for meaningful changes
The system should log meaningful lifecycle events like:
* reminder log created
* due date changed
* reminder log deactivated
* chase task created
* chase task deactivated because reminder became invalid
Not every low-level evaluation needs a visible activity line, but key state changes should be auditable.

Where the anchor milestone is time-sensitive, reminder due dates must be calculated using the milestone event date rather than the completion timestamp.
6.36 Reminder logic must be auditable
The system must preserve enough information to explain why a reminder exists, what date was used to calculate it, and when its state changed.
6.37 Reminder rules may target milestone outcomes or operational actions
Reminder Rules may point either to milestone-related follow-up logic or to operational actions such as exchange-readiness checks.
6.38 When a reminder rule becomes invalid, the live Reminder Log is deactivated
If a Reminder Rule no longer applies to a transaction because the underlying conditions are no longer met, the existing live Reminder Log should be deactivated rather than left active.
6.39 Reminder generation runs from a scheduled backend process
Reminder rule evaluation must run from a recurring backend job rather than relying on front-end activity.
6.40 Reminder generation evaluates active transactions against current rule conditions
The scheduled process evaluates each relevant active transaction against milestone state, not required state, event dates, exchange readiness, and reminder rule definitions.
6.41 Reminder Logs are created or updated before Chase Tasks are generated
Reminder Logs act as the system layer between reminder logic and human task creation. Chase Tasks are created from eligible Reminder Logs rather than directly from milestone logic.
6.42 Valid rules create or update live Reminder Logs
If a rule becomes newly valid, the system creates a live Reminder Log. If the rule remains valid and source conditions change, the existing live Reminder Log is updated.
6.43 Invalid rules deactivate live Reminder Logs
If a rule no longer applies because source conditions are no longer met, the live Reminder Log is deactivated.
6.44 Chase Tasks are created only from due active Reminder Logs
A Chase Task is only created when its related Reminder Log is active, due, and does not already have an equivalent open live task.
6.45 Duplicate open Chase Tasks must be prevented
For a given live reminder state, the system should normally allow only one open Chase Task at a time.
6.46 Overdue state is derived
Overdue state is derived from active or open status and due date rather than stored as a separate uncontrolled state.
6.47 Rule invalidation should also deactivate related open task paths
Where a live Reminder Log is deactivated because a rule no longer applies, related open Chase Tasks should also be deactivated or marked no longer relevant in an auditable way.
6.48 On hold transactions pause reminder generation
Transactions marked on hold should pause reminder generation unless intentionally resumed by later logic or manual action.
6.49 Meaningful reminder lifecycle changes must be auditable
The system must record meaningful reminder lifecycle changes such as creation, due date recalculation, deactivation, and task creation.
